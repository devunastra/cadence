# Spec: Cancel Scheduled Callbacks

## Summary

Add a new **Scheduled Callbacks** tab to the Follow-ups page that lists every AI callback currently queued in n8n's AI Callback data table. Studio staff can cancel any pending callback to stop the AI agent from making the call — e.g., when they've already handled the lead manually.

**Client intent (verbatim):**
> "I want the ability to manually cancel followup calls. Like if the client asked to be called back tomorrow at 1pm, but one of the studio staff calls them manually, we should be able to cancel the followup."

---

## Background: How AI Callbacks Work Today

1. During a Retell call, the lead says "call me back tomorrow at 1pm"
2. The Retell agent POSTs to the `schedule_ai_callback` webhook in n8n's **Voice AI Functions** workflow
3. The webhook parses the time and inserts a row into the **n8n "AI Callback" data table** (ID: `9U0GXNR5uRUTWUPy`)
4. Every 30 minutes, the `AI Callback Trigger` (n8n schedule trigger) queries for rows where `callback_time <= now AND called_at IS EMPTY`
5. For each pending row, it calls `Retell create-phone-call` to make the outbound call
6. After calling, it stamps `called_at` with the current timestamp (so the row is skipped on future runs)

### n8n AI Callback Data Table Schema

| Column | Type | Description |
|--------|------|-------------|
| `id` | auto | Row ID (auto-generated by n8n) |
| `first_name` | string | Lead's first name |
| `last_name` | string | Lead's last name |
| `phone_number` | string | Lead phone (E.164 format) |
| `email` | string | Lead email |
| `dance_interest` | string | What dance they're interested in |
| `reason` | string | Reason for inquiry |
| `callback_time` | datetime | When to call back |
| `called_at` | datetime | Timestamp when call was made (empty = pending) |

The n8n data table is the **source of truth** for the callback queue. Cancelling = stamping `called_at` so the trigger skips the row on its next 30-minute poll.

### Relationship to call_reviews

The `call_reviews` table (Supabase) is populated **after** a call by GPT analysis. It flags `callback_requested = true` when the transcript shows the lead asked for a callback — this drives the existing **Callback Requests** tab on the Follow-ups page (analytical view, unchanged by this feature).

The two systems are independent:
- A `call_reviews.callback_requested` row may exist without a matching n8n row (the AI didn't schedule one)
- An n8n row may exist without a `call_reviews` row (the call is queued but hasn't happened yet, so no review)

**This feature operates on the n8n table directly.** It does not touch `call_reviews`.

## Users

| Role | What they see | What they can do |
|------|---------------|------------------|
| `super_admin` | All pending callbacks across all studios (matched to a lead) | Cancel any |
| `studio_owner` | Pending callbacks matched to a lead in their studio(s) | Cancel any in their studios |
| `studio_staff` | Same as `studio_owner` | Cancel any in their studios |

Studio isolation is enforced **in the server action**, not at the DB layer (the data lives in n8n, not Supabase). The match works as follows:
- For each n8n pending row, normalize `phone_number` to E.164
- Look up `leads.studio_id` via `leads.phone` (normalized)
- **Drop any row that has no matching lead in Supabase** — orphans are hidden from everyone, including super_admin. Every visible row must exist in both n8n and `leads`.
- For studio members, further filter to rows whose `studio_id` is in their `studio_users` memberships
- For super_admin, keep all matched rows (no studio filter)

## Edge Cases

| # | Category | Scenario | Expected Behavior | Severity |
|---|----------|----------|-------------------|----------|
| 1 | Matching | Phone format drift between n8n (`+15551234567`) and `leads.phone` (`(555) 123-4567`) | Normalize both sides to E.164 before matching: strip all non-digits, prefix `+1` if 10 digits. | Medium |
| 2 | Matching | n8n row's phone matches no lead in Supabase (orphan row) | **Hidden from everyone** including super_admin. Visible row must exist in both n8n and `leads`. | Low |
| 3 | Matching | n8n row's phone matches multiple leads in different studios | Use the first match. (In production a phone should be unique per studio; cross-studio duplicates indicate a data issue surfaced separately.) | Low |
| 4 | Concurrency | 30-min n8n trigger fires between list-fetch and cancel-fetch on the user's screen | Cancel webhook returns `rows_updated: 0` (row's `called_at` is already stamped). Info toast: "Callback was already made by the AI agent." Row removed from view. | Medium |
| 5 | Concurrency | Two staff members open the tab and cancel the same row at the same time | First cancel wins; second sees `rows_updated: 0` and the same "already cancelled" info toast. | Low |
| 6 | Network | n8n webhook times out or returns 5xx | Error toast: "Failed to cancel callback — please retry." Row stays in view. | Medium |
| 7 | Network | n8n list-fetch fails on initial tab load | Show error state + Retry button. Don't break the page. | Medium |
| 8 | Auth | Header secret mismatch (misconfigured env) | n8n returns 401. Server action surfaces "Cancel webhook not configured." | High |
| 9 | Auth | Session expired mid-cancel | Server action returns Unauthorized → redirect to login. | Medium |
| 10 | Browser | Double-click cancel button | Button disabled on first click; prevents duplicate requests. | Low |
| 11 | UX | User cancels then immediately leaves the page | Optimistic removal already applied; no rollback needed. | Low |
| 12 | Data | n8n row has missing fields (e.g., null email) | Render `—` placeholder in `var(--color-text-muted)`. | Low |

## Affected Layers

### DB: None
This feature operates on the n8n data table only. No Supabase schema changes, no migration, no new column.

### RLS: None
Same reason — no Supabase tables touched. Studio isolation is enforced in the server action via lead-phone matching.

### Server actions: `app/actions.ts`
- **New:** `fetchScheduledCallbacks()` — calls n8n webhook with `{ action: 'list' }`, filters rows by user's studio membership via lead-phone match, returns enriched `ScheduledCallback[]`
- **New:** `cancelScheduledCallback(n8nRowId: string)` — calls n8n webhook with `{ action: 'cancel', n8n_row_id }`, returns `{ success, rowsUpdated }`

### API routes: None
n8n webhook is called from the server action (server-side fetch).

### n8n: New combined webhook workflow
- **One** workflow with **two webhook trigger nodes** (cleaner than a single Switch-routed webhook — each path is independently debuggable)
- **`POST /scheduled-callbacks-list { }`** → Get rows from AI Callback table where `called_at IS NULL`, return `{ rows: [...] }`
- **`POST /scheduled-callbacks-cancel { phone_number }`** → Update WHERE phone_number = $phone_number AND called_at IS NULL SET called_at = now(); return `{ success: true, rowsUpdated: number }`
- Both webhooks share **one** Header Auth credential (header name `X-Callbacks-Secret`)
- Created via n8n MCP tools — **NEW workflow, not modifying VAF or production**
- **Cancel filter MUST use `matchType: "allConditions"` (AND)** — without it, the filter degrades to OR semantics and a single cancel would stamp every pending row in the queue.
- **Why phone-based instead of id-based:** the n8n Data Table node does not expose the auto-generated `id` column in its filter schema. Attempting `keyName: "id"` causes n8n to silently strip the field on save, leaving an incomplete filter. The fix is to filter on `phone_number` (a user-defined column). Per edge case #3, cancelling all pending rows for a phone is acceptable behavior — leads have at most one pending callback in practice.
- Cancel operation is **idempotent** via compound filter (`phone_number = $phone AND called_at IS NULL`) — re-cancelling an already-stamped phone matches 0 rows (`rowsUpdated: 0`).
- **Built:** workflow ID `DrMdkkkCZBZTu3OS`, name `AMLS Scheduled Callbacks Webhook (Joshua)`

## Data Flow

### List (on tab open + on focus refresh)

```
User opens Scheduled Callbacks tab (or window regains focus)
  → fetchScheduledCallbacks() server action fires
    → n8n webhook: POST { action: 'list' } with auth header
      → n8n queries AI Callback table for rows WHERE called_at IS NULL
      → n8n returns [{ id, first_name, last_name, phone_number, email, dance_interest, reason, callback_time }, ...]
    → Server action normalizes all phone_numbers to E.164
    → Supabase: SELECT id, studio_id, phone FROM leads
                WHERE phone IN (normalized_phones)
                  AND studio_id IN (user_studio_ids)
    → Match n8n rows to leads by normalized phone
    → Filter:
        - super_admin: keep all rows (matched + orphans)
        - studio_owner/staff: keep only matched rows in their studios
    → Sort by callback_time asc
    → Return ScheduledCallback[]
  → Table renders
```

### Cancel

```
User clicks PhoneOff icon on a row
  → Confirmation dialog opens with lead name
  → User confirms
  → cancelScheduledCallback(n8nRowId) server action fires
    → n8n webhook: POST { action: 'cancel', n8n_row_id } with auth header
      → n8n updates row WHERE id = n8n_row_id AND called_at IS NULL SET called_at = now()
      → n8n returns { success: true, rows_updated: 0 | 1 }
    → Server action returns { success, rowsUpdated }
  → UI:
      - rows_updated = 1: optimistically remove row, success toast "Callback cancelled"
      - rows_updated = 0: optimistically remove row, info toast "Callback was already made by the AI agent"
      - error: row stays, error toast
```

## Phone Normalization

Both sides must agree on E.164 format (`+1XXXXXXXXXX`) for matching to work.

**Normalization function (server action):**
```ts
function normalizePhone(raw: string | null): string | null {
  if (!raw) return null
  const digits = raw.replace(/\D/g, '')
  if (digits.length === 10) return `+1${digits}`
  if (digits.length === 11 && digits.startsWith('1')) return `+${digits}`
  if (raw.startsWith('+') && digits.length >= 10) return `+${digits}`
  return null   // unrecognized — treat as no match
}
```

Apply to both n8n row's `phone_number` and `leads.phone` before comparing.

## Out of Scope (MVP)

- **Trigger manually** (fire the callback now instead of waiting for the 30-min trigger) — explicitly cut from scope
- **Reschedule callback** (edit `callback_time` in n8n) — separate feature
- **Bulk cancel** (multi-select rows) — not needed yet
- **Audit trail of who cancelled** — no `cancelled_by`/`cancelled_at` recorded anywhere; cancellation is fire-and-forget
- **Cancel from CallDetailDrawer or Call History** — Scheduled Callbacks tab is the only entry point
- **Persistent filters on this tab** — no filter UI in v1
- **Polling for live updates** — manual refresh + tab focus is enough; no 30s poll
- **Adding `studio_id` to the n8n data table** — defer until multi-studio rollout

## Known Limitations / Future Work

- **Single-studio assumption:** AMLS runs on one studio in production today. The phone-based studio match works in this mode but is fragile when studios share phone numbers (unlikely but possible). When AMLS adds studio #2, consider adding `studio_id` to the n8n table and updating VAF's `schedule_ai_callback` to populate it.
- **No real-time visibility into the 30-min auto-trigger firing:** if the AI agent calls a queued lead while staff is viewing the tab, staff won't see the row disappear until the next manual refresh or tab-focus event. This is acceptable for v1.
- **Action labels in the modal:** the "Cancel" button on the modal both means "cancel the dialog" and "cancel the callback" depending on which button. UX copy should be: dismiss button says "Keep Callback" or "Back", confirm button says "Cancel Callback".
